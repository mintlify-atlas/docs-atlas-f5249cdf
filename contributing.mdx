## Before you start

OCA contributions follow the standard GitHub fork-and-pull-request model combined with pre-commit hooks that enforce code quality automatically. Read this page end-to-end before opening your first pull request.

<Info>
  By submitting code to OCA you agree that it will be published under the license declared in the module's `__manifest__.py` (AGPL-3.0 or LGPL-3.0). See [Licenses](#licenses) below.
</Info>

## Contribution workflow

<Steps>
  <Step title="Fork and clone">
    Fork [OCA/web](https://github.com/OCA/web) on GitHub, then clone your fork and add the upstream remote:

    ```bash
    git clone https://github.com/<your-handle>/web.git
    cd web
    git remote add upstream https://github.com/OCA/web.git
    git fetch upstream
    ```
  </Step>
  <Step title="Create a branch from 18.0">
    Always branch from the `18.0` branch, which is the active development branch:

    ```bash
    git checkout -b 18.0-feat-my-improvement upstream/18.0
    ```

    Use a descriptive branch name that starts with the Odoo series (`18.0-`).
  </Step>
  <Step title="Install pre-commit">
    Pre-commit runs all code quality checks locally before each commit, matching the same checks that CI runs on pull requests.

    ```bash
    pip install pre-commit
    pre-commit install
    ```

    To run the checks manually on all files:

    ```bash
    pre-commit run --all-files
    ```
  </Step>
  <Step title="Make your changes">
    Edit or add files. Follow the [code style rules](#code-style) described below. If you are adding a new addon, see [Adding a new addon](#adding-a-new-addon).
  </Step>
  <Step title="Commit">
    Pre-commit hooks run automatically on `git commit`. They will fix many issues (formatting, trailing whitespace) and then fail so you can review the auto-fixes before committing again:

    ```bash
    git add .
    git commit -m "[ADD] web_my_feature: initial implementation"
    ```

    OCA commit message convention: `[ADD]`, `[FIX]`, `[IMP]`, `[REF]`, `[REM]`, or `[MIG]` followed by the addon name and a short description.
  </Step>
  <Step title="Push and open a pull request">
    ```bash
    git push origin 18.0-feat-my-improvement
    ```

    Open a pull request against the `OCA/web:18.0` branch on GitHub. The PR description should explain **what** changed and **why**. Reference any related issues with `Fixes #123`.
  </Step>
</Steps>

## Code style

The repository enforces style automatically through `.editorconfig` and the pre-commit hooks.

### Editor settings (`.editorconfig`)

| File type | Setting | Value |
|---|---|---|
| All supported types | Charset | `utf-8` |
| All supported types | Line endings | `lf` |
| Python, XML, CSS, JS | Indent | 4 spaces |
| JSON, YAML, RST, MD | Indent | 2 spaces |
| All supported types | Trailing whitespace | trimmed |
| All supported types | Final newline | inserted |

Configure your editor to respect `.editorconfig` automatically (most editors support this natively or via a plugin).

### Pre-commit hooks

The following tools run on every commit:

<AccordionGroup>
  <Accordion title="Prettier (v3.3.3) with @prettier/plugin-xml">
    Formats CSS, HTML, JS, JSON, Markdown, SCSS, TOML, TypeScript, XML, and YAML files. Runs with `--write` so it auto-fixes issues in place. You must stage the auto-fixed files and commit again.
  </Accordion>
  <Accordion title="ESLint (v9.12.0)">
    Lints JavaScript files. Uses `eslint-plugin-jsdoc` for JSDoc validation. Runs with `--fix` to auto-correct safe issues. Configuration lives in `eslint.config.cjs`.
  </Accordion>
  <Accordion title="Ruff (v0.6.8)">
    Fast Python linter and formatter. Replaces Flake8 and Black for Python files. Runs `ruff check --fix` followed by `ruff format`.
  </Accordion>
  <Accordion title="pylint-odoo (v9.1.3)">
    Odoo-specific pylint checks. Two passes run: one with optional checks (exit-zero) and one with mandatory checks that fail the commit on violations. Configuration is in `.pylintrc` and `.pylintrc-mandatory`.
  </Accordion>
  <Accordion title="OCA maintainer-tools hooks">
    - `oca-gen-addon-readme`: regenerates `README.rst` from fragment files in `readme/` when sources change.
    - `oca-fix-manifest-website`: ensures `website` in every manifest points to `https://github.com/OCA/web`.
    - `oca-gen-external-dependencies`: updates `requirements.txt` from manifest `external_dependencies`.
    - `oca-update-pre-commit-excluded-addons`: keeps the exclude list in `.pre-commit-config.yaml` accurate.
  </Accordion>
  <Accordion title="pre-commit-hooks (v4.6.0)">
    Standard hygiene: trailing whitespace, end-of-file newline, debug statements, encoding pragma removal, case-conflict detection, merge-conflict markers, symlink checks, XML validation, and LF line-ending enforcement.
  </Accordion>
  <Accordion title="whool (v1.3)">
    Generates `setup/` packaging metadata for each addon so it can be installed via pip.
  </Accordion>
</AccordionGroup>

## Adding a new addon

### Directory structure

Each addon lives in a top-level directory named after the technical module name. A minimal addon looks like this:

```text
web_my_feature/
├── __init__.py
├── __manifest__.py
├── readme/
│   ├── DESCRIPTION.md
│   ├── INSTALL.md          # optional
│   ├── CONFIGURE.md        # optional
│   └── CHANGELOG.md        # optional
├── static/
│   └── src/
│       └── js/
│           └── my_feature.esm.js
└── views/
    └── my_feature_views.xml
```

<Note>
  Do not hand-write `README.rst`. The `oca-gen-addon-readme` pre-commit hook generates it automatically from the `readme/` fragment files.
</Note>

### Manifest template

Base your manifest on this template, taken from the existing `web_notify` addon:

```python
# Copyright <year> <Your Name / Company>
# License LGPL-3.0 or later (http://www.gnu.org/licenses/lgpl).

{
    "name": "Web My Feature",
    "summary": "Short one-line description of what the addon does",
    "version": "18.0.1.0.0",
    "license": "LGPL-3",
    "author": "Your Name, Odoo Community Association (OCA)",
    "development_status": "Alpha",   # Alpha | Beta | Production/Stable
    "website": "https://github.com/OCA/web",
    "depends": ["web"],
    "assets": {
        "web.assets_backend": [
            "web_my_feature/static/src/js/my_feature.esm.js",
        ],
    },
    "installable": True,
}
```

Key manifest fields:

| Field | Required | Notes |
|---|---|---|
| `name` | Yes | Human-readable display name |
| `summary` | Yes | One sentence shown in the Apps list |
| `version` | Yes | Must start with `18.0.` |
| `license` | Yes | `AGPL-3` or `LGPL-3` |
| `author` | Yes | Include `Odoo Community Association (OCA)` |
| `website` | Yes | Must be `https://github.com/OCA/web` (enforced by hook) |
| `depends` | Yes | At minimum `["web"]` |
| `installable` | Yes | Set to `True` |
| `development_status` | Recommended | `Alpha`, `Beta`, or `Production/Stable` |
| `external_dependencies` | If needed | Python/bin packages; hook updates `requirements.txt` |

### External Python dependencies

If your addon depends on a third-party Python package, declare it in the manifest:

```python
"external_dependencies": {
    "python": ["my_package"],
},
```

The `oca-gen-external-dependencies` hook will automatically add the package to the repository's `requirements.txt`.

## Licenses

The repository is licensed under AGPL-3.0. Individual addons may use **AGPL-3.0** or **LGPL-3.0**:

- Use **LGPL-3.0** (`"license": "LGPL-3"`) when the addon is a library or reusable widget that other (possibly proprietary) addons may depend on.
- Use **AGPL-3.0** (`"license": "AGPL-3"`) for standalone feature addons.

Add the appropriate copyright header to every Python file:

<Tabs>
  <Tab title="LGPL-3.0">

## FAQ

<AccordionGroup>
  <Accordion title="What Python version is required?">
    The pre-commit configuration specifies `python: python3` with no lower bound. In practice you need the same Python version that your Odoo 18.0 installation uses. Odoo 18.0 supports Python 3.10 through 3.12. The Node.js version used for Prettier and ESLint is pinned to **22.9.0** in `.pre-commit-config.yaml`.
  </Accordion>
  <Accordion title="How are tests run?">
    OCA uses Odoo's built-in test runner. To run tests for a single addon:

    ```bash
    odoo \
      --test-enable \
      --stop-after-init \
      -d test_db \
      -i web_notify
    ```

    `test-requirements.txt` contains `odoo_test_helper`, which provides utilities for writing Odoo unit tests. CI runs the full test suite via GitHub Actions on every pull request (see the **Build Status** badge on the repository).
  </Accordion>
  <Accordion title="Can I contribute an addon that depends on Odoo Enterprise?">
    No. All addons in this repository must be installable on Odoo Community Edition. Addons that would conflict with Enterprise modules should declare `"excludes": ["web_enterprise"]` in their manifest, as `web_dark_mode` and `web_responsive` do.
  </Accordion>
  <Accordion title="How do I update the README for my addon?">
    Do not edit `README.rst` directly. Write or update the Markdown fragment files in the addon's `readme/` directory (e.g., `readme/DESCRIPTION.md`). The `oca-gen-addon-readme` pre-commit hook regenerates `README.rst` automatically when those source files change.
  </Accordion>
  <Accordion title="How do I add a translation?">
    OCA manages translations through [Weblate](https://translation.odoo-community.org/engage/web-18-0/). Do not add a `i18n/en.po` file — the `en-po-files` pre-commit hook will reject it. Translations for other languages are exported from Weblate and merged back by bots.
  </Accordion>
  <Accordion title="My pre-commit hook keeps failing — how do I debug it?">
    Run the failing hook in isolation:

    ```bash
    pre-commit run <hook-id> --all-files
    ```

    For example, to run only Ruff:

    ```bash
    pre-commit run ruff --all-files
    ```

    Add `--verbose` for more detail. Many hooks auto-fix issues and then exit non-zero so you can review changes before re-staging them.
  </Accordion>
</AccordionGroup>
